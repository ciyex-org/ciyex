## Key Architectural Changes

### Authentication Flow

#### Before (Local Auth Only)
```
User → Email/Password → Local DB → JWT Token → Org/Facility Assignment
```

#### After (Dual Auth Support)
```
User → Choose Auth Method
  ├─ Local: Email/Password → Local DB → JWT Token → Org/Facility
  └─ Keycloak: Username/Password → Keycloak → Access Token → Groups
```

### Data Model Changes

#### Tenant/Organization Model (Local Auth)
- User belongs to Organizations
- Organizations have Facilities
- Users have Roles per Organization

#### Groups Model (Keycloak Auth)
- User belongs to Groups
- Groups represent organizations, roles, or facilities
- Groups stored as flat array in localStorage
- Example groups: `["org-hospital-a", "role-doctor", "facility-main"]`

### Token Management

#### Local Auth
- Custom JWT generated by backend
- Contains user info, org, and facility data
- Stored in localStorage as `token`

#### Keycloak Auth
- Access token from Keycloak (standard OAuth2 JWT)
- Contains user info, groups, and roles
- Stored in localStorage as `token`
- Additional `refresh_token` available for token renewal

## Security Enhancements

1. **OAuth2 Standard Compliance**: Using industry-standard OAuth2/OIDC protocols
2. **Centralized User Management**: Users managed in Keycloak
3. **Group-Based Access Control**: Flexible group membership instead of rigid tenant structure
4. **Token Validation**: JWT tokens validated against Keycloak's public keys
5. **SSO Ready**: Infrastructure in place for Single Sign-On

## Backward Compatibility

The integration maintains full backward compatibility:
- Local authentication still works

- Both auth methods can coexist
- No breaking changes to existing functionality
